## Условие

- Написать вьюер каркасной 3D-модели с использованием структурного подхода
- Приложение с графическим интерфейсом, загружающее модель из файла (список точек и соединяющих их рёбер) и отображающее её как каркас.
- Возможности, которые должны быть реализованы:
  - Загрузка модели из файла (формат: список точек + список рёбер). Не хардкодьте путь - сделайте кнопку/диалог в GUI для выбора файла.
  - Преобразования в пространстве: перенос, масштабирование, вращение Масштабирование и вращение - относительно произвольной точки.
  - Отрисовка каркаса (точки и рёбра) в окне приложения.
- Сохранение модели в файл - опционально.

## Требования 

Основная цель лабы - показать владение принципами структурного проектирования и аккуратной организации кода. Конкретные требования:
- Чёткое разделение на домены:
    - Интерфейсный домен (GUI) - отвечает исключительно за ввод/вывод и отрисовку.
    - Прикладной домен - отвечает за хранение модели, расчёты преобразований и выдачу данных для отрисовки.
    - Логика прикладного домена должна быть полностью изолирована от GUI: GUI только вызывает прикладные функции и отображает их результат. Интерфейсный домен преподавателя не интересует - он смотрит прикладную часть.
- Чёткая точка входа в прикладной домен (например, единая функция-обработчик/диспетчер).
- Чёткое разделение параметров на входные, изменяемые и выходные.
- Функции должны иметь понятную декомпозицию и разделение обязанностей.
- Функция принимает не более трёх параметров.
- Для параметров, в которые функция вносит изменения и которые могут остаться в некорректном состоянии при ошибке, создавайте временную копию и работайте с ней; в случае успеха копируйте результат в выходной параметр. (Для выходных параметров это не обязательно.)
- Функция не должна изменять входные параметры при невозможности выполнения
- Не перекладывать ответственность за ресурсы на другие функции: если функция открыла файл - она же его и закрывает; если выделила память - она должна освободить в случае ошибки или вернуть владельца обратно.
- Соблюдать уровни абстракции: одна функция не должна одновременно работать со структурой как целым и с её внутренними полями. Разделяйте выделение памяти/инициализацию структуры и работу с её полями. Работа с "полями полей" (доступ к полям вложенных структур из внешней функции) - недопустима.
- Проверки параметров - обязательно делать в самом начале функции, до выполнения какой-либо основной логики. Это ранний выход (первый из максимум двух логических выходов).
- Максимум два логических выхода из функции: один - ранний при проверке параметров (СТРОГО ДО ЛЮБОЙ ЛОГИКИ), второй - в конце функции (возврат результата/кода ошибки). Если по ходу функции возникает ошибка, возвращайте код ошибки через все уровни (не игнорируйте).
- Не более трёх уровней вложенности внутри функции.
- Чёткая декомпозиция: каждая функция - одна ответственность.
- Использование goto строго запрещено
